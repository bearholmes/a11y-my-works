# 유스케이스 명세서

## 개요

이 문서는 업무 보고 시스템의 사용자 시나리오와 유스케이스를 정의합니다.

## 액터 (Actors)

### 주요 액터
- **일반 사용자 (직원)**: 자신의 업무를 보고하고 조회하는 사용자
- **팀장**: 팀원의 업무를 조회하고 승인하는 사용자
- **관리자**: 시스템 전체를 관리하고 설정하는 사용자
- **시스템**: 자동화된 작업을 수행하는 시스템

### 역할별 권한
| 역할 | 권한 키 | 설명 |
|------|---------|------|
| 관리자 | PERM_01~10 | 모든 권한 (대시보드, 업무보고, 프로젝트, 서비스, 사용자 등) |
| 팀장 | PERM_01, 02, 10 | 대시보드, 업무보고, 로그 조회 |
| 직원 | PERM_01, 02 | 대시보드, 본인 업무보고만 |

### UC-001: 회원가입

**ID**: UC-001
**이름**: 회원가입
**액터**: 미가입 사용자
**설명**: 새로운 사용자가 시스템에 가입한다
**우선순위**: 높음

#### 전제 조건
- 사용자가 유효한 이메일 주소를 보유하고 있다
- 시스템이 정상 작동 중이다

#### 주요 흐름
1. 사용자가 로그인 페이지에서 "회원가입" 버튼을 클릭한다
2. 시스템이 회원가입 폼을 표시한다
3. 사용자가 정보를 입력한다:
   - 이메일 주소
   - 비밀번호 (최소 8자)
   - 이름
   - 계정 ID (사용자 고유 식별자)
4. 시스템이 입력 정보를 검증한다:
   - 이메일 형식 확인
   - 비밀번호 강도 확인
   - 중복 이메일 확인
5. 시스템이 Supabase Auth에 사용자를 생성한다
6. 시스템이 members 테이블에 프로필을 생성한다 (기본 역할: 직원)
7. 시스템이 확인 이메일을 전송한다
8. 시스템이 성공 메시지를 표시한다
9. 사용자가 자동으로 로그인된다

#### 대안 흐름
**4a. 이메일이 이미 등록되어 있는 경우**
- 시스템이 "이미 등록된 이메일입니다" 오류 메시지를 표시한다
- 3단계로 돌아간다

**4b. 비밀번호가 너무 약한 경우**
- 시스템이 "비밀번호는 최소 8자 이상이어야 합니다" 오류 메시지를 표시한다
- 3단계로 돌아간다

**6a. 프로필 생성 실패**
- 시스템이 Auth 계정을 생성했으나 프로필 생성에 실패한다
- 시스템이 로그에 오류를 기록한다
- 사용자는 로그인 후 프로필을 수동으로 완성해야 한다

#### 예외 흐름
**E1. 네트워크 오류**
- 시스템이 "네트워크 오류가 발생했습니다. 다시 시도해주세요" 메시지를 표시한다
- 유스케이스가 종료된다

**E2. Supabase 서버 오류**
- 시스템이 "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요" 메시지를 표시한다
- 오류를 로그에 기록한다

#### 후속 조건
- 사용자 계정이 생성되고 로그인 상태가 된다
- 사용자가 대시보드 페이지로 이동한다

#### 참조 구현
- `src/components/LoginForm.tsx:48` - 회원가입 폼
- `src/hooks/useAuth.ts:48` - signUp 함수
- `src/services/api.ts:138` - createMemberProfile

### UC-002: 로그인

**ID**: UC-002
**이름**: 로그인
**액터**: 등록된 사용자
**설명**: 사용자가 시스템에 로그인한다
**우선순위**: 높음

#### 전제 조건
- 사용자가 이미 회원가입을 완료했다
- 사용자가 이메일과 비밀번호를 알고 있다

#### 주요 흐름
1. 사용자가 로그인 페이지에 접속한다
2. 시스템이 로그인 폼을 표시한다
3. 사용자가 이메일과 비밀번호를 입력한다
4. 사용자가 "로그인" 버튼을 클릭한다
5. 시스템이 Supabase Auth로 인증을 시도한다
6. Supabase가 인증을 확인하고 세션을 생성한다
7. 시스템이 사용자 정보를 로드한다 (members 테이블)
8. 시스템이 전역 상태(Jotai)에 사용자 정보를 저장한다
9. 시스템이 대시보드 페이지로 리다이렉트한다

#### 대안 흐름
**5a. 이메일 또는 비밀번호가 틀린 경우**
- 시스템이 "이메일 또는 비밀번호가 올바르지 않습니다" 오류 메시지를 표시한다
- 3단계로 돌아간다

**5b. 계정이 비활성화된 경우**
- 시스템이 "계정이 비활성화되었습니다. 관리자에게 문의하세요" 메시지를 표시한다
- 유스케이스가 종료된다

**7a. members 테이블에 프로필이 없는 경우**
- 시스템이 기본 프로필을 자동 생성한다
- 8단계로 진행한다

#### 예외 흐름
**E1. 네트워크 오류**
- 시스템이 "네트워크 오류가 발생했습니다" 메시지를 표시한다
- 3단계로 돌아간다

#### 후속 조건
- 사용자가 로그인 상태가 된다
- 세션이 로컬 스토리지에 저장된다
- 사용자가 보호된 페이지에 접근할 수 있다

#### 참조 구현
- `src/components/LoginForm.tsx:1` - 로그인 폼
- `src/hooks/useAuth.ts:38` - signIn 함수
- `src/components/AppRouter.tsx:10` - ProtectedRoute

### UC-003: 업무 보고 작성

**ID**: UC-003
**이름**: 업무 보고 작성
**액터**: 직원 (로그인된 사용자)
**설명**: 사용자가 수행한 업무를 시스템에 보고한다
**우선순위**: 매우 높음 (핵심 기능)

#### 전제 조건
- 사용자가 로그인되어 있다
- 사용자가 PERM_02(업무보고) 쓰기 권한을 보유하고 있다
- 시스템에 최소 1개 이상의 프로젝트/서비스가 등록되어 있다

#### 주요 흐름
1. 사용자가 네비게이션에서 "업무 보고 작성" 메뉴를 클릭한다
2. 시스템이 업무 보고 작성 폼을 표시한다
3. 시스템이 비용 그룹 목록을 로드하여 드롭다운에 표시한다
4. 사용자가 비용 그룹을 선택한다
5. 시스템이 선택된 비용 그룹에 속한 서비스 목록을 로드한다
6. 사용자가 서비스를 선택한다
7. 시스템이 선택된 서비스에 속한 프로젝트 목록을 로드한다
8. 사용자가 프로젝트를 선택한다
9. 사용자가 업무 정보를 입력한다:
   - 업무 날짜 (기본값: 오늘)
   - 업무 유형
   - 업무명 (필수)
   - 업무 세부 내용
   - 업무 URL
   - 작업 시간 (분 단위)
   - 시작 시간
   - 종료 시간
   - 플랫폼
   - 버전
10. 사용자가 "제출" 버튼을 클릭한다
11. 시스템이 폼 데이터를 검증한다 (Zod 스키마):
    - 필수 필드 확인
    - 업무 시간 유효성 확인 (0 < 시간 ≤ 1440분)
    - 날짜 형식 확인
12. 시스템이 현재 로그인된 사용자의 member_id를 확인한다
13. 시스템이 tasks 테이블에 새 레코드를 생성한다
14. 시스템이 TanStack Query 캐시를 무효화한다
15. 시스템이 "업무 보고가 등록되었습니다" 성공 메시지를 표시한다
16. 시스템이 업무 목록 페이지로 리다이렉트한다

#### 대안 흐름
**4a. 사용자가 비용 그룹을 변경하는 경우**
- 시스템이 서비스와 프로젝트 선택을 초기화한다
- 5단계로 진행한다

**6a. 사용자가 서비스를 변경하는 경우**
- 시스템이 프로젝트 선택을 초기화한다
- 7단계로 진행한다

**11a. 업무 시간이 24시간(1440분)을 초과하는 경우**
- 시스템이 "업무 시간은 24시간을 초과할 수 없습니다" 오류 메시지를 표시한다
- 9단계로 돌아간다

**11b. 필수 필드가 비어있는 경우**
- 시스템이 해당 필드에 "필수 항목입니다" 오류 메시지를 표시한다
- 9단계로 돌아간다

**11c. 시작 시간이 종료 시간보다 늦은 경우**
- 시스템이 "시작 시간은 종료 시간보다 빨라야 합니다" 오류 메시지를 표시한다
- 9단계로 돌아간다

#### 예외 흐름
**E1. 데이터베이스 연결 오류**
- 시스템이 "서버 연결에 실패했습니다" 메시지를 표시한다
- 사용자가 다시 시도할 수 있다

**E2. 권한 부족**
- 시스템이 "업무 보고 작성 권한이 없습니다" 메시지를 표시한다
- 유스케이스가 종료된다

**E3. 세션 만료**
- 시스템이 "세션이 만료되었습니다. 다시 로그인해주세요" 메시지를 표시한다
- 시스템이 로그인 페이지로 리다이렉트한다

#### 후속 조건
- 새로운 업무 보고가 데이터베이스에 저장된다
- 사용자가 업무 목록에서 방금 작성한 보고를 확인할 수 있다
- 대시보드 통계가 업데이트된다

#### 비기능 요구사항
- **응답 시간**: 폼 제출 후 2초 이내에 결과 표시
- **가용성**: 99.9% 가동률
- **데이터 무결성**: 트랜잭션 보장 (Supabase RLS)

#### 참조 구현
- `src/pages/TaskForm.tsx:1` - 업무 보고 폼 UI
- `src/services/api.ts:86` - createTask API
- `src/types/database.ts:93` - Task 타입 정의

### UC-004: 업무 보고 조회

**ID**: UC-004
**이름**: 업무 보고 조회
**액터**: 직원, 팀장, 관리자
**설명**: 사용자가 업무 보고 목록을 조회하고 필터링한다
**우선순위**: 높음

#### 전제 조건
- 사용자가 로그인되어 있다
- 사용자가 PERM_02(업무보고) 읽기 권한을 보유하고 있다

#### 주요 흐름
1. 사용자가 네비게이션에서 "업무 보고" 메뉴를 클릭한다
2. 시스템이 업무 보고 목록 페이지를 표시한다
3. 시스템이 기본 조건으로 업무 목록을 조회한다:
   - 최근 20건
   - 생성일 기준 내림차순
   - 사용자 권한에 따라 필터링 (직원: 본인 업무만, 팀장/관리자: 전체)
4. 시스템이 업무 목록을 테이블 형태로 표시한다:
   - 날짜, 업무명, 프로젝트, 서비스, 작업 시간, 작성자
5. 사용자가 필터 옵션을 선택한다 (선택적):
   - 날짜 범위 (시작일 ~ 종료일)
   - 프로젝트
   - 서비스
   - 비용 그룹
   - 작성자 (팀장/관리자만)
   - 키워드 검색
6. 사용자가 "검색" 버튼을 클릭한다
7. 시스템이 필터 조건에 맞는 업무 목록을 조회한다
8. 시스템이 업데이트된 목록을 표시한다

#### 대안 흐름
**3a. 조회 결과가 없는 경우**
- 시스템이 "등록된 업무가 없습니다" 메시지를 표시한다
- 유스케이스가 종료된다

**7a. 필터링 결과가 없는 경우**
- 시스템이 "검색 결과가 없습니다" 메시지를 표시한다
- 5단계로 돌아간다

**8a. 사용자가 페이지네이션 버튼을 클릭하는 경우**
- 시스템이 다음/이전 페이지 데이터를 로드한다
- 8단계로 돌아간다

**8b. 사용자가 업무 항목을 클릭하는 경우**
- 시스템이 UC-005(업무 보고 수정) 또는 상세 조회 페이지로 이동한다

#### 예외 흐름
**E1. 권한 부족**
- 시스템이 "업무 보고 조회 권한이 없습니다" 메시지를 표시한다
- 대시보드로 리다이렉트한다

#### 후속 조건
- 사용자가 업무 목록을 확인할 수 있다
- 필터 조건이 URL 쿼리 파라미터에 저장된다 (북마크 가능)

### UC-005: 업무 보고 수정

**ID**: UC-005
**이름**: 업무 보고 수정
**액터**: 직원, 관리자
**설명**: 사용자가 기존 업무 보고를 수정한다
**우선순위**: 중간

#### 전제 조건
- 사용자가 로그인되어 있다
- 사용자가 PERM_02(업무보고) 쓰기 권한을 보유하고 있다
- 수정하려는 업무 보고가 존재한다
- 본인이 작성한 업무 보고이거나 관리자 권한을 보유하고 있다

#### 주요 흐름
1. 사용자가 업무 목록에서 수정할 업무를 클릭한다
2. 시스템이 업무 보고 수정 페이지로 이동한다 (`/tasks/edit/:id`)
3. 시스템이 해당 업무의 기존 데이터를 로드한다
4. 시스템이 폼에 기존 데이터를 채운다
5. 사용자가 수정할 필드를 변경한다
6. 사용자가 "수정" 버튼을 클릭한다
7. 시스템이 변경된 데이터를 검증한다 (UC-003과 동일한 규칙)
8. 시스템이 소유권을 확인한다:
   - 본인 작성 또는 관리자 권한
9. 시스템이 tasks 테이블의 레코드를 업데이트한다
10. 시스템이 updated_at 필드를 현재 시각으로 갱신한다
11. 시스템이 "업무 보고가 수정되었습니다" 메시지를 표시한다
12. 시스템이 업무 목록 페이지로 리다이렉트한다

#### 대안 흐름
**3a. 업무 보고가 존재하지 않는 경우**
- 시스템이 "해당 업무를 찾을 수 없습니다" 오류 메시지를 표시한다
- 업무 목록 페이지로 리다이렉트한다

**8a. 소유권이 없는 경우 (다른 사람의 업무를 수정하려는 경우)**
- 시스템이 "이 업무를 수정할 권한이 없습니다" 오류 메시지를 표시한다
- 유스케이스가 종료된다

#### 예외 흐름
**E1. 동시 수정 충돌**
- 다른 사용자가 동일한 업무를 동시에 수정한 경우
- 시스템이 "다른 사용자가 먼저 수정했습니다. 페이지를 새로고침하세요" 메시지를 표시한다
- updated_at 타임스탬프로 감지

#### 후속 조건
- 업무 보고가 수정된 내용으로 업데이트된다
- 수정 이력이 updated_at 필드에 기록된다

#### 참조 구현
- `src/pages/TaskForm.tsx:1` - 수정 폼 (생성과 동일 컴포넌트)

### UC-010: 프로젝트 관리 (관리자)

**ID**: UC-010
**이름**: 프로젝트 관리
**액터**: 관리자
**설명**: 관리자가 프로젝트를 등록, 수정, 삭제한다
**우선순위**: 중간

#### 전제 조건
- 사용자가 로그인되어 있다
- 사용자가 PERM_03(프로젝트 관리) 쓰기 권한을 보유하고 있다

#### 주요 흐름

##### 10.1 프로젝트 등록
1. 관리자가 "프로젝트 관리" 메뉴를 클릭한다
2. 시스템이 프로젝트 목록을 표시한다
3. 관리자가 "새 프로젝트 추가" 버튼을 클릭한다
4. 시스템이 프로젝트 등록 폼을 표시한다
5. 관리자가 프로젝트 정보를 입력한다:
   - 프로젝트명 (필수)
   - 서비스 선택
   - 플랫폼
   - 버전
   - 업무 타입
   - 메모

7. 시스템이 입력 정보를 검증한다
8. 시스템이 projects 테이블에 새 레코드를 생성한다
9. 시스템이 "프로젝트가 등록되었습니다" 메시지를 표시한다
10. 시스템이 프로젝트 목록을 새로고침한다

##### 10.2 프로젝트 수정
1. 관리자가 프로젝트 목록에서 수정할 프로젝트를 클릭한다
2. 시스템이 기존 데이터를 채운 수정 폼을 표시한다
3. 관리자가 정보를 수정한다
4. 시스템이 변경사항을 저장한다

##### 10.3 프로젝트 비활성화
1. 관리자가 프로젝트의 "비활성화" 버튼을 클릭한다
2. 시스템이 is_active를 false로 설정한다
3. 해당 프로젝트는 업무 보고 작성 시 드롭다운에 표시되지 않는다

#### 대안 흐름
**7a. 중복된 프로젝트명이 존재하는 경우**
- 시스템이 "이미 존재하는 프로젝트명입니다" 경고 메시지를 표시한다
- 5단계로 돌아간다

**10.3a. 프로젝트에 연결된 업무 보고가 있는 경우**
- 시스템이 "이 프로젝트는 X건의 업무와 연결되어 있습니다" 경고를 표시한다
- 관리자가 확인하면 비활성화 진행
